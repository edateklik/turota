# Turota Coding Agent Instructions

## 1. Project overview

Turota is a team-owned local discovery and trip-planning product. The repository currently contains:

- An ASP.NET Core/.NET backend organized as modular application, domain, and infrastructure projects.
- A Python FastAPI AI recommendation service.
- PostgreSQL/PostGIS integration and database migrations.
- Docker Compose configuration.
- Unit, integration, load, and AI-service tests.
- Development tools and project documentation.

A Flutter application will be added later under the top-level `mobile/` directory. These instructions apply to all coding agents working in this repository, with particular emphasis on keeping mobile work isolated from existing team-owned systems.

## 2. Repository boundaries

Treat the existing backend, AI service, database code, migrations, tests, Docker configuration, tools, and documentation as owned by other team members.

The following existing areas are outside the default scope of mobile work:

- `src/`
- `ai-service/`
- `tests/`
- `tools/`
- `docs/`
- `compose.yaml`
- Root-level .NET solution, build, package, and SDK configuration files
- Existing environment templates and configuration files

Do not change these areas unless the user explicitly requests the exact change. A mobile requirement does not implicitly authorize backend or infrastructure work. Preserve all existing team members' work, including uncommitted changes.

## 3. Mobile application scope

The Flutter application will live entirely under `mobile/`. Until the user explicitly asks for project creation, do not create the Flutter project or the `mobile/` directory.

Mobile work may include:

- Android and iOS application code.
- Flutter screens, native Flutter widgets, assets, themes, and localization.
- Mobile networking, authentication, secure storage, navigation, and state management.
- Mobile unit, widget, golden, and integration tests.
- Mobile-specific developer documentation inside `mobile/`.

Flutter must consume the published ASP.NET API contracts. It must not call the internal FastAPI service directly and must not access PostgreSQL directly.

## 4. Allowed and forbidden file modifications

Unless the user explicitly expands the scope, mobile implementation may modify only files under `mobile/`.

Allowed after Flutter project creation is explicitly requested:

- Files created by `flutter create` under `mobile/`.
- Handwritten Dart code, Flutter tests, assets, and mobile configuration under `mobile/`.
- Generated mobile files when regenerated by their owning tool.

Forbidden without an explicit user request:

- Backend endpoints, request/response contracts, domain models, migrations, or application services.
- FastAPI models, algorithms, settings, or endpoints.
- Existing backend, AI, integration, load, or database tests.
- Dockerfiles, `compose.yaml`, CI/CD, root build configuration, or repository documentation.
- Any existing team member's unrelated or uncommitted work.

Never modify backend API contracts merely to make a mobile screen easier to implement. Report missing or unsuitable contracts and design the mobile code around confirmed contracts until the backend owner resolves them.

Create no secrets or real environment files. Never commit passwords, access or refresh tokens, signing material, API keys, service keys, private URLs, or real `.env` files. Use documented placeholders and ignored local configuration.

Do not manually edit generated files unless it is necessary, the owning generation workflow cannot express the change, and the reason is documented. Prefer regenerating them with the appropriate tool.

## 5. Flutter architecture rules

Use a feature-first architecture with clear presentation, domain, and data boundaries.

- UI code must not call HTTP, SignalR, secure storage, or platform plugins directly.
- Repositories expose domain-oriented operations and hide transport details.
- Data sources own HTTP, realtime, persistence, and platform integration.
- DTOs model wire contracts; domain models represent application concepts. Do not reuse DTOs as mutable UI state.
- Map DTOs to domain models at the data boundary.
- Keep cross-feature code in `core/` or `shared/` only when at least two features genuinely use it.
- Avoid speculative abstractions and broad base classes.
- Dependencies should point inward: presentation to domain abstractions, and data implementations toward those abstractions.
- Platform-specific code must remain behind Dart interfaces where practical.

## 7. UI and Stitch implementation rules

Stitch HTML/CSS designs are visual and structural references only.

- Recreate designs as native Flutter widgets.
- Do not use `WebView`, `HtmlElementView`, embedded browsers, inline HTML renderers, or bundled Stitch HTML/CSS.
- Do not copy browser layout assumptions directly; adapt them to Flutter constraints and platform conventions.
- Extract colors, typography, spacing, radii, shadows, imagery, hierarchy, and interaction intent from the reference.
- Support safe areas, keyboard insets, text scaling, accessibility semantics, and common phone widths.
- Avoid fixed screen coordinates and fragile pixel-perfect positioning.
- Use responsive constraints, scrolling, and flexible layouts.
- Preserve the intended visual hierarchy while prioritizing usable native behavior.
- Do not invent backend-driven data or interactions. Use explicit loading, empty, error, and unavailable states where contracts are missing.

## 8. Reusable components and design system rules

- Define semantic design tokens for colors, typography, spacing, radii, elevation, and animation timing.
- Use the application theme instead of repeating raw styles.
- Extract a reusable widget when it appears more than once or represents a stable product pattern.
- Keep one-off layout fragments local to the screen.
- Reusable widgets must accept domain-neutral inputs and must not reach into feature state implicitly.
- Provide consistent buttons, fields, cards, loading indicators, error states, empty states, and screen scaffolds.
- Build accessible tap targets and meaningful semantic labels.
- Do not add a second component library or styling system without explicit agreement.
- Reuse existing assets and components before creating alternatives.

## 9. State management and navigation principles

- Use one state-management approach consistently across the application; do not mix competing patterns within features.
- Keep ephemeral widget state local and application/business state in feature controllers.
- Represent loading, success, empty, and failure states explicitly.
- Make asynchronous actions idempotent where repeated taps or realtime duplicate delivery is possible.
- Keep navigation declarative and centralized under `app/router/`.
- Routes should use typed arguments or stable identifiers, not entire mutable response objects.
- Authentication state controls protected-route redirects.
- Deep links and back navigation must not bypass authentication or required onboarding.
- Realtime events accelerate refreshes; REST remains the source used to reconcile state after reconnects.

Choose or add state-management and navigation packages only when the user authorizes dependency installation or project setup.

## 10. Networking, API models, authentication, and error handling rules

- Flutter calls the ASP.NET API only. Never expose or use the FastAPI service key in the mobile application.
- Keep API base URLs in environment-specific mobile configuration; do not scatter URLs through feature code.
- Account for platform development hosts: Android Emulator commonly uses `10.0.2.2`, while iOS Simulator can use `localhost`.
- Send JWTs through `Authorization: Bearer <token>`.
- Store tokens only with an approved secure-storage mechanism, never plain preferences, logs, fixtures, screenshots, or source code.
- The current backend has no refresh-token contract. Handle expiration by clearing authenticated state and requiring login; do not fabricate refresh behavior.
- Model UUIDs, dates, times, nullable fields, and string enums deliberately.
- Treat unknown enum values safely so a server addition does not crash the application.
- Parse RFC 7807 responses and branch on stable `errorCode`, not localized `title` or `detail`.
- Preserve `traceId` for diagnostics without showing technical details by default.
- Do not log authorization headers, tokens, passwords, personal data, or full sensitive payloads.
- Apply reasonable timeouts and distinguish connectivity, timeout, authorization, validation, conflict, rate-limit, and server failures.
- Recommendation creation is asynchronous. Support `202`, status polling, terminal states, and duplicate SignalR events keyed by `runId`.
- Reconcile state through REST after SignalR reconnects.
- Keep DTO field names faithful to the OpenAPI/backend contract. Put presentation formatting in mappers or view models.

## 11. Code quality and naming conventions

- Follow `dart format`, `flutter analyze`, and the repository's eventual lint configuration.
- Use `lower_snake_case.dart` filenames, `UpperCamelCase` types, and `lowerCamelCase` members.
- Name pages by product purpose, such as `TripDetailPage`, not by design-export labels.
- Suffix wire types with `RequestDto` or `ResponseDto` when ambiguity exists.
- Name repository abstractions by capability and implementations clearly.
- Prefer immutable models and state.
- Keep widgets focused; split large build methods into meaningful private or reusable widgets.
- Avoid unexplained constants, duplicated literals, forced null assertions, and broad exception catches.
- Comments should explain constraints or intent, not restate code.
- Remove dead code, unused assets, debug prints, and temporary design experiments before completion.

## 12. Testing and validation expectations

Validation must be proportional to the change and remain within the authorized scope.

- Add unit tests for DTO parsing, mappers, repositories, controllers, and error mapping where behavior exists.
- Add widget tests for meaningful screen states and user interactions.
- Add golden tests only after fonts, rendering environment, and intentional tolerances are stable.
- Add integration tests for critical flows such as authentication and recommendation-to-trip navigation when the backend contract supports them.
- Mock at system boundaries, not every internal class.
- Include success, loading, empty, validation, authorization, rate-limit, connectivity, and retry states as relevant.
- Run formatting, static analysis, and the smallest relevant test set before handoff.
- Do not start Docker, backend services, or the AI service unless the user explicitly authorizes it.
- Do not run migrations.
- Report commands run, results, and any validation that could not be performed.

## 13. Git branch, commit, and pull request workflow

Mobile development is performed on `feature/mobile-eda-foundation`.

Before modifying any file, agents must run read-only checks for:

```bash
git status --short --branch
git branch --show-current
```

- Confirm the current branch and inspect the worktree before editing.
- Do not switch branches, discard changes, reset files, or overwrite another contributor's work without explicit authorization.
- Stop and report if the requested change would overlap unrelated uncommitted work.
- Implement and commit each mobile screen separately so review and rollback remain focused.
- Each screen commit must include only that screen, its necessary reusable components, and its relevant tests.
- Use concise, imperative commit messages with a clear mobile scope.
- Do not create commits, push branches, or open/update pull requests unless the user explicitly requests that action.
- Before a requested commit, inspect the diff and ensure it contains no secrets, generated noise, or unrelated changes.
- Pull requests should describe implemented states, backend dependencies, screenshots or recordings where appropriate, tests performed, and known contract gaps.

## 14. Safe working procedure for coding agents

For every task:

1. Read this file and any more specific `AGENTS.md` governing the target path.
2. Inspect `git status --short --branch` and `git branch --show-current`.
3. Read the relevant existing mobile code, tests, API contracts, and design references before proposing edits.
4. Confirm that the requested work is within `mobile/` or is otherwise explicitly authorized.
5. Identify backend contract gaps without changing backend contracts.
6. Plan the smallest coherent implementation for one screen or one infrastructure capability.
7. Preserve unrelated and uncommitted work.
8. Modify only necessary files and avoid manual edits to generated output.
9. Format, analyze, and run the smallest relevant tests when permitted.
10. Review `git diff` and `git status` for accidental changes, secrets, and scope creep.
11. Report the exact files changed, validation performed, and remaining risks.
12. Commit only when explicitly requested, and keep each screen in its own commit.

When blocked by a missing contract, asset, design decision, credential, or permission, do not guess in a way that changes product behavior. Implement an honest unavailable state if appropriate, or stop and request the missing decision.

## 15. Definition of done for each mobile screen

A mobile screen is done only when:

- It is implemented with native Flutter widgets and does not embed Stitch HTML/CSS.
- It matches the approved reference's hierarchy, typography, spacing, color, and interaction intent.
- It uses the shared design system and existing reusable components appropriately.
- It works at supported phone widths, honors safe areas and keyboard insets, and scrolls where necessary.
- It has meaningful accessibility semantics, readable text scaling, and adequate touch targets.
- Loading, success, empty, offline/network error, server error, and retry states are handled where applicable.
- Authentication, validation, conflict, and rate-limit errors are handled according to confirmed API contracts.
- It does not expose secrets or sensitive data in code or logs.
- Business logic and networking are outside widget build methods and follow the feature architecture.
- Relevant unit and widget tests are added and passing.
- Formatting and static analysis pass for the changed scope.
- No backend, AI service, database, Docker, unrelated test, tool, or documentation file was changed without explicit authorization.
- The final diff contains only the screen, necessary shared components, and related tests.
- Known API or design gaps are documented in the handoff.
- When commits are requested, the screen is committed separately with a focused message.
